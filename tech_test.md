## Тестовое задание Million Agents

Время на выполнение: 3-4 часа
Стек: Python 3.10+, FastAPI, PostgreSQL, Redis

Контекст 

Есть упрощённый сервис управления задачами (task tracker). Пользователи создают задачи, назначают исполнителей, задачи проходят через статусы. Нужно реализовать бэкенд-часть.

Часть 1. Проектирование схемы данных (без кода)

Дана следующая бизнес-логика:

— Есть проекты, у каждого проекта есть владелец и участники
— Есть задачи, каждая задача принадлежит проекту
— У задачи есть: название, описание, приоритет (low/medium/high/critical), статус, автор, исполнитель
— Задача проходит через статусы: `created` -> `in_progress` -> `review` -> `done`
— Но есть ограничения на переходы:
  - Из `created` можно перейти только в `in_progress` или `cancelled`
  - Из `in_progress` — в `review` или `created` (вернуть)
  - Из `review` — в `done` или `in_progress` (вернуть на доработку)
  - Из `done` и `cancelled` переходов нет
— Нужно хранить историю изменений статусов (кто, когда, из какого в какой)

Задание:
1. Нарисуй ER-диаграмму как удобно в любом формате, может быть текстом, ссылкой на draw.io, miro и тд
2. Напиши SQL-миграцию (CREATE TABLE) для PostgreSQL
3. Объясни словами: почему выбрал именно такую структуру? Какие индексы добавил бы и почему?

Часть 2. Реализация API

Реализуй FastAPI-приложение со следующими эндпоинтами:

Обязательные эндпоинты

POST   /tasks/              — создать задачу
GET    /tasks/              — список задач (с фильтрацией по статусу и приоритету)
GET    /tasks/{id}          — получить задачу
PATCH  /tasks/{id}/status   — сменить статус задачи
GET    /tasks/{id}/history  — история изменений статуса задачи

Требования

— Используй SQLAlchemy (async) + Alembic для миграций
— Pydantic-схемы для запросов и ответов
— Валидация переходов статусов (см. правила выше). При невалидном переходе — вернуть 400 с понятным сообщением
— Пагинация на списке задач
— Для простоты: автора/исполнителя можно передавать как `user_id` (int) в теле запроса, без авторизации

Бонус (необязательно)

— Фильтрация задач по исполнителю
— Сортировка по дате создания / приоритету
— Docker Compose для запуска (app + postgres)

Часть 3. Вопросы на понимание

Если сталкивался с чем-то похожим на практике — приведи пример из своего опыта (на каком проекте, что случилось, как решал). 3.1. Конкурентность

Два пользователя одновременно пытаются сменить статус одной задачи:
— Пользователь A: `in_progress` -> `review`
— Пользователь B: `in_progress` -> `created`

Оба читают текущий статус `in_progress`, оба прошли валидацию. Что произойдёт? Как бы ты это решил?

3.2. Дебаг по логам

Приходит баг-репорт: "Задача #4521 застряла в статусе `review`, хотя я точно нажимал кнопку 'Готово'".

Какие шаги ты предпримешь для расследования? Какие логи/данные посмотришь? Какие гипотезы проверишь первыми?

3.3. Проектирование

Завтра приходит продукт и говорит: "Нам нужно добавить комментарии к задачам, и чтобы при смене статуса можно было оставить комментарий-причину".

Как бы ты это реализовал? Что поменяешь в текущей схеме? Нужна ли отдельная таблица?

3.4. Нагрузка

Сервис вырос, задач стало 10 миллионов. Эндпоинт `GET /tasks/` стал тормозить. Какие конкретные шаги предпримешь? (Не абстрактно "добавить кэш", а конкретно — что закэшируешь, какие запросы оптимизируешь, какие индексы посмотришь.)
