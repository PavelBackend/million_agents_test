## Решение 3 части:

3.1. Конкурентность

Я бы решал подобную гонку через добавление фильтра по полю status, то есть примерно такой запрос: UPDATE tasks SET status=... WHERE id=... AND status='in_progress'. В случае не обновления записи возвращал бы 409 ошибку conflict.
Второй вариант более блокирующий на большой нагрузке - это собственно использование лока select ... for update

3.2. Дебаг по логам

Я бы начал с восстановления таймлайна по данной задаче: кто и когда нажимал “готово”, какая ручка дергалась, какой payload и какой ответ вернулся клиенту. Смотрю логи приложения, историю изменений статуса в БД, и логи фоновых задач/очередей, если смена статуса асинхронная. Могу предположить такие проблемы: запрос не дошёл из-за ошибки сети, дошёл но упал на валидации, дошёл и ответил 200, но потом другой процесс перезаписал статус. В Arevord мы так искали пропавшие изменения: по сorrelation id в ELK быстро находили, что запрос дошёл, например, но обновление не применилось из-за повторной обработки сообщения

3.3. Проектирование

Для комментария-причины при смене статуса в текущей схеме мне достаточно использовать уже существующую таблицу task_status_history, так как в ней уже есть поле comment. Если же продукт хочет полноценные комментарии к задаче вне смены статуса, тогда я бы добавил отдельную таблицу task_comments. При смене статуса делал бы один атомарный сценарий в транзакции: проверить допустимость перехода, обновить статус задачи с защитой от гонок и записать событие в task_status_history с комментарием. Так мы сохраняем и историю переходов, и причину изменения

3.4. Нагрузка

Сначала измерю, медленный ли сам запрос к Postgres с помощью EXPLAIN (ANALYZE, BUFFERS). Далее, конкретно для GET /tasks/: включаю строгую пагинацию cursor pagination вместо offset, а детали через отдельную ручку для конкретной таски отдельным GET /tasks/{id}.
По индексам - смотрел бы реальные фильтры и сортировки из самых частых запросов, например project_id + status + updated_at.
Из кэша я бы начал с результатов самых частых списочных запросов, например первую страницу задач проекта по статусу или список активных задач пользователя, также отдельно кэшировал бы агрегаты вроде количества задач по статусам
